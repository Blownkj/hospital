# План тестирования — Hospital IS

## Подготовка среды

```fish
cd ~/hospital/hospital
docker-compose up -d
docker-compose exec web php database/seeds.php
```

Проверить доступность:
- Приложение: http://localhost:8000/
- phpMyAdmin: http://localhost:8081/ (user / password)

Получить учётные данные:
```fish
docker-compose exec web php database/seeds.php 2>&1 | grep -E "@"
```

Синтаксис PHP всех файлов:
```fish
docker-compose exec web find /var/www/html/src -name "*.php" \
  -exec php -l {} \; 2>&1 | grep -v "No syntax errors"
```

Быстрая проверка HTTP-кодов:
```fish
for url in "/" "/doctors" "/services" "/about" "/faq" "/contact" "/articles" "/login" "/register"
    curl -s -o /dev/null -w "%{http_code} $url\n" "http://localhost:8000$url"
end
```

## Блок 6 — Безопасность

### 6.1 Изоляция ролей

| # | Попытка | Ожидаемый результат |
|---|---------|---------------------|
| 6.1.1 | Пациент → `GET /admin/dashboard` | 403 или redirect |
| 6.1.2 | Пациент → `GET /doctor/dashboard` | 403 или redirect |
| 6.1.3 | Врач → `GET /admin/dashboard` | 403 или redirect |
| 6.1.4 | Врач → `GET /patient/dashboard` | 403 или redirect |
| 6.1.5 | Гость → `GET /patient/book` | Redirect на `/login` |
| 6.1.6 | Авторизованный пользователь → `GET /login` | Redirect на свой дашборд |

### 6.2 CSRF

```bash
# POST без токена
curl -X POST http://localhost:8000/patient/appointments/cancel \
  -d "appointment_id=1"
```

Ожидается: flash-ошибка "Недействительный CSRF-токен", redirect — **не 500**.

### 6.3 Сессия истекла → отправка формы

1. Войти, открыть `GET /patient/book`
2. Удалить куки в браузере (DevTools → Application → Cookies → Clear)
3. Нажать "Записаться"

Ожидается: `CsrfMiddleware` перехватывает, redirect с flash-ошибкой — **не белый экран**.

### 6.4 IDOR — доступ к чужим данным

```bash
# Отменить чужую запись
curl -X POST http://localhost:8000/patient/appointments/cancel \
  -b cookies_patient1.txt \
  -d "appointment_id=<ID_ЧУЖОЙ_ЗАПИСИ>&csrf_token=<VALID_TOKEN>"

# Распечатать чужой визит
curl -b cookies_patient1.txt \
  http://localhost:8000/patient/visit/<ЧУЖОЙ_VISIT_ID>/print
```

Ожидается: 403 или тихое игнорирование — данные не изменяются.

### 6.5 XSS

В любое текстовое поле (имя, bio, протокол) ввести:
```
<script>alert(1)</script>
```
Открыть страницу, где это отображается. Скрипт **не должен выполниться** — `View::e()` экранирует через `htmlspecialchars`.

### 6.6 SQL Injection

| Где | Что ввести | Ожидаемый результат |
|-----|-----------|---------------------|
| `/doctors/{id}` | `1' OR '1'='1` | 404, не утечка данных |
| Поле email при входе | `' OR 1=1 --` | Flash-ошибка входа |
| Любое поле формы | `'; DROP TABLE users; --` | Ошибка валидации; PDO prepared statements защищают |

### 6.7 HTTP-заголовки безопасности

```bash
curl -I http://localhost:8000/
```

Проверить наличие:
```
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'; ...
```

### 6.8 Пароль не попадает в логи

1. Попробовать войти с неверным паролем `supersecret123`
2. Открыть лог:

```fish
docker-compose exec web cat /var/www/html/logs/app-(date +%Y-%m-%d).log
```

Ожидается: запись `Login failed` есть, строки `supersecret123` в файле **нет**.
