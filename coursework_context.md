# Контекст курсовой работы: Информационная система медицинской клиники

**Тема:** Разработка веб-приложения для автоматизации работы медицинской клиники  
**Стек:** PHP 8.1, MySQL 8.0, Apache, Docker  
**Паттерн:** MVC (кастомная реализация без фреймворка)

---

## 1. Описание проекта

Разработанная система — это веб-приложение для автоматизации деятельности многопрофильной медицинской клиники. Приложение реализует полный операционный цикл: от записи пациента к врачу до ведения протокола приёма, выдачи назначений и публикации отзывов.

Система построена по принципу монолитного серверного приложения (Server-Side Rendering): весь HTML формируется на сервере, JavaScript используется только для улучшения пользовательского взаимодействия. Такой подход обеспечивает простоту развёртывания и сопровождения при полноте функциональности.

Приложение разделено на три изолированных раздела по ролям пользователей: **пациент**, **врач**, **администратор** — каждый со своим набором страниц, прав доступа и функций.

Ключевые характеристики проекта:
- **Кастомный MVC-стек** — все инфраструктурные компоненты (маршрутизатор, шаблонизатор, работа с сессиями, логирование, пагинация, валидация) написаны с нуля.
- **Строгая типизация** — `declare(strict_types=1)` во всех PHP-файлах, PHP 8.1 Backed Enums, readonly-свойства, типизированные сигнатуры методов.
- **Комплексная безопасность** — защита от CSRF, XSS, SQL-инъекций, тайминг-атак, перехвата сессии, rate limiting на критичных маршрутах.
- **Docker-окружение** — три контейнера (PHP+Apache, MySQL 8.0, phpMyAdmin), автоматическая инициализация схемы БД при первом запуске.

## 2. Назначение системы

Система решает задачи автоматизации работы частной медицинской клиники, заменяя ручное ведение журналов записи, бумажные протоколы приёмов и телефонную запись пациентов.

**Для пациентов** система предоставляет: самостоятельную онлайн-запись к врачу и на лабораторные анализы, доступ к личной медицинской карте с историей визитов и назначениями, возможность оставлять отзывы о качестве обслуживания.

**Для врачей** система предоставляет: расписание приёмов на текущий день, рабочий стол для ведения приёма (заполнение протокола, добавление назначений), историю обращений пациента и статистику собственной работы.

**Для администратора** система предоставляет: управление составом врачей и их расписаниями, контроль всех записей пациентов с возможностью отмены и переноса, модерацию отзывов, управление прайс-листом услуг и каталогом лабораторных анализов, аналитический дашборд с графиками и экспортом данных в CSV.

## 4. Архитектура

### 4.1 Паттерн MVC

Приложение реализует классический трёхслойный паттерн **Model–View–Controller**:

- **Model (Модель)** — данные и бизнес-логика. Разделена на два уровня:
  - *Репозитории* (`src/Repositories/`) — инкапсулируют все SQL-запросы к базе данных.
  - *DTO-модели* (`src/Models/`) — иммутабельные объекты `User` и `DoctorProfile`, гидратируемые из результатов БД.
  - *Сервисы* (`src/Services/`) — оркестрируют бизнес-логику, объединяя несколько репозиториев.

- **View (Представление)** — PHP-шаблоны в директории `views/`. Шаблон получает данные через `View::render('имя', $data)`, который делает `extract($data)` и подключает PHP-файл. Весь вывод данных экранируется через `View::e()`.

- **Controller (Контроллер)** — классы в `src/Controllers/`, наследующие `BaseController`. Принимают HTTP-запрос, читают входные данные, вызывают сервисы/репозитории, передают результат во View или выполняют редирект.

### 4.2 Слоистая архитектура и жизненный цикл запроса

```
HTTP-запрос
    ↓
public/index.php — точка входа (загрузка .env, заголовки безопасности, сессия, регистрация маршрутов)
    ↓
Router::dispatch() — сопоставление URL с маршрутом, извлечение параметров пути
    ↓
Глобальный Middleware — CsrfMiddleware (для всех POST)
    ↓
Групповой Middleware — AuthMiddleware::requireRole() (для /patient/*, /doctor/*, /admin/*)
    ↓
Маршрутный Middleware — RateLimitMiddleware (для /login, /register), OwnerMiddleware (для печати визита)
    ↓
Controller::method() — обработка запроса
    ↓
Service — бизнес-логика (AuthService, AppointmentService, DoctorService)
    ↓
Repository — запросы к БД через PDO
    ↓
View::render() — рендеринг PHP-шаблона
    ↓
HTTP-ответ
```

### 4.3 Middleware-архитектура

Маршрутизатор поддерживает три уровня middleware:

1. **Глобальные** — применяются ко всем запросам. Пример: `CsrfMiddleware::handle()`.
2. **Групповые** — применяются к группе маршрутов с общим префиксом. Пример: `requireRole('patient')` для всех `/patient/*`.
3. **Маршрутные** — применяются только к конкретному маршруту. Пример: `RateLimitMiddleware` для `POST /login`, `OwnerMiddleware` для `GET /patient/visit/{id}/print`.

## 5. Frontend

### 5.1 Подход к шаблонизации

Фронтенд системы построен на **PHP-шаблонах с серверным рендерингом**. Каждая страница состоит из:
- Layout-файла (`views/layout/`), подключаемого через `require` в начале и конце шаблона.
- Основного шаблона страницы, содержащего HTML с минимальным PHP для вывода данных.
- Партиалов (`views/partials/`) — переиспользуемых фрагментов.

Шаблоны следуют правилу: **вся логика в контроллерах и сервисах, шаблон только отображает**. Переменные попадают в шаблон через `View::render('путь/шаблон', ['key' => $value])`, который выполняет `extract()`.

### 5.2 CSS-архитектура

CSS-файл (`public/css/main.css`) организован через **CSS Cascade Layers** — современный стандарт управления каскадом:

```css
@layer reset, components, utilities;
```

Слои выстроены по приоритету: `utilities` всегда перекрывает `components`, `components` перекрывает `reset`. Это исключает конфликты специфичности.

**Дизайн-токены** — все значения вынесены в CSS Custom Properties (`--space-*`, `--color-*`, `--text-*`, `--radius-*`). Цвета определены в цветовом пространстве `oklch()` для точного управления насыщенностью и светлотой.

**Типографика** — три Google Fonts:
- *Inter Tight* — основной шрифт интерфейса.
- *Instrument Serif* — акцентные заголовки публичной части.
- *JetBrains Mono* — числовые данные и метки.

Fluid-типографика через `clamp()` позволяет плавно масштабировать размеры без медиа-запросов.

**BEM-нейминг** — `.block`, `.block__element`, `.block--modifier`. Примеры: `.btn--primary`, `.card__title`, `.badge--success`, `.doctor-card__avatar`.

**Адаптивность** — три брейкпойнта: mobile < 768px, tablet 768px, desktop 1024px+. Мобильное меню реализовано без JavaScript — CSS-only через `checkbox` + `~` (general sibling selector).

**Доступность** — `skip-link` для быстрого перехода к контенту, стили `focus-visible`, медиа-запрос `prefers-reduced-motion` для отключения анимаций по системным настройкам.

### 5.3 JavaScript

JavaScript минималистичен — **Vanilla JS**, около 62 строк в `public/js/app.js`. Никаких фреймворков. Функциональность:

- **Выбор временного слота** — по клику на `.slot-btn` выделяет кнопку, записывает время в скрытое поле формы и показывает блок подтверждения.
- **Подтверждение действий** — формы с атрибутом `data-confirm` показывают `window.confirm()` перед отправкой.
- **Аккордеон FAQ** — открытие/закрытие ответов с ARIA-атрибутами `aria-expanded`.
- **Перенос записи (admin)** — toggle-скрытие формы переноса.
- **Выбор анализа** — отображение деталей выбранного лабораторного теста.

Отдельный файл `admin-charts.js` подключает **Chart.js** (CDN) и рисует два графика на дашборде администратора — линейный (динамика записей за 14 дней) и столбчатый (топ-5 врачей). Данные передаются через `data-*` атрибуты `<canvas>`.

### 5.4 Иконки

50+ иконок из библиотеки **Lucide** встроены прямо в PHP-код как SVG-пути. Функция `icon($name, $size, $class)` выводит нужный SVG с `aria-hidden="true"`. Преимущества: нет внешних HTTP-запросов за иконками, полный контроль размера и цвета через CSS.

### 5.5 Компоненты интерфейса

Переиспользуемые партиалы в `views/partials/`:

- `flash.php` — уведомления четырёх типов: success, error, warning, info. Каждый тип имеет SVG-иконку и CSS-цвет. ARIA-атрибут `role="alert"`.
- `icon.php` — функция-библиотека SVG-иконок.
- `status-badge.php` — цветная метка статуса записи (маппинг: `pending → жёлтый`, `confirmed/completed → зелёный`, `in_progress → синий`, `cancelled → красный`).
- `doctor-card.php` — карточка врача со специализацией, рейтингом и биографией.
- `pagination.php` — навигация по страницам с кнопками prev/next и диапазоном номеров.
- `empty-state.php` — заглушка для пустых списков.

## 6. Backend

### 6.1 Инфраструктурный слой (src/Core/)

**Router** (`src/Core/Router.php`) — кастомный маршрутизатор. Поддерживает GET и POST. Параметры пути `{id}` компилируются в регулярные выражения и передаются в контроллер как именованные аргументы. Поддерживает группировку маршрутов с префиксом и middleware-цепочкой.

**Session** (`src/Core/Session.php`) — управление сессиями, CSRF-токенами и flash-сообщениями. CSRF-токен генерируется через `bin2hex(random_bytes(32))` (криптографически стойкий), хранится в сессии, сравнивается через `hash_equals()` для защиты от тайминг-атак. Flash-сообщения читаются и удаляются за одно обращение.

**View** (`src/Core/View.php`) — шаблонизатор. Метод `View::e($value)` — обязательное экранирование вывода через `htmlspecialchars()`. Вспомогательные методы: `initials($name)` — заглавные буквы имени для аватаров, `stars($rating)` — строка из символов рейтинга.

**Logger** (`src/Core/Logger.php`) — ротируемый файловый логгер без внешних зависимостей. Записи вида `[2026-05-09 14:23:01] INFO: User logged in {"user_id":42}`. Ротация: файлы старше 30 дней автоматически удаляются. Уровни: `info()`, `warning()`, `error()`.

**Paginator** (`src/Core/Paginator.php`) — расчёт параметров пагинации: смещение `offset`, диапазон номеров страниц, флаги `hasPrev()`/`hasNext()`.

**Validator** (`src/Core/Validator.php`) — набор статических методов: `email()`, `password()` (минимум 8 символов), `phone()`, `dateInFuture()`, `time()`, `nonEmpty()`.

### 6.2 Контроллеры

Все контроллеры наследуют `BaseController`, который предоставляет вспомогательные методы: `validateCsrf()`, `redirectWith($path, $type, $msg)`, `redirectBack()`, `input($key)`, `requireOwner($condition)`.

**AuthController** — регистрация, вход, выход. При ошибках регистрации форма перерендеривается с ошибками и сохранёнными значениями полей. После успешного входа происходит перенаправление в кабинет по роли.

**PatientController** (486 строк) — кабинет пациента: дашборд с предстоящими записями, многошаговая запись к врачу, запись на анализы, просмотр истории, профиль, отзывы, печать визита.

**DoctorController** (243 строки) — кабинет врача: дашборд с расписанием на день, страница приёма, управление протоколом и назначениями, история приёмов.

**AdminController** (596 строк) — административная панель: дашборд с аналитикой, управление записями (пагинация, фильтры, отмена, перенос), расписания врачей, модерация отзывов, CRUD врачей, управление услугами и анализами, экспорт CSV.

**PublicController** — публичный сайт: главная страница (список врачей, специализации, последние отзывы), каталог врачей с поиском, страница врача, услуги, статьи, FAQ, контакты.

### 6.3 Сервисы

**AuthService** — бизнес-логика аутентификации. Проверяет пароль через `password_verify()`, обновляет устаревший хеш (`password_needs_rehash()`), выполняет `session_regenerate_id(true)` при входе (защита от session fixation), проверяет активность врача, записывает все события в лог.

**AppointmentService** — логика записи:
- `getWorkingDays(doctorId)` — строит список ближайших рабочих дней врача из 60 дней вперёд, с учётом расписания и исключений (отпусков).
- `getSlots(doctorId, date)` — генерирует временные слоты от start_time до end_time с шагом slot_duration_min, помечает занятые и прошедшие.
- `book(data)` — тройная защита от двойного бронирования (подробнее — в разделе «Инженерные решения»).

**DoctorService** — жизненный цикл приёма: начало, сохранение протокола, завершение. Управление назначениями. Все изменения статуса выполняются транзакционно. Нарушение бизнес-правил выбрасывает кастомные исключения `DomainException` или `ForbiddenException`.

### 6.4 Репозитории

Все репозитории наследуют `BaseRepository` с методами `findById()`, `findAll()`, `count()`, `delete()`, `transaction()`.

**UserRepository** — работа с таблицей `users`: поиск по email, создание пользователя, смена пароля, проверка активности врача.

**AppointmentRepository** (330 строк) — центральный репозиторий: занятые слоты, блокировка слота (`SELECT FOR UPDATE`), создание записи, записи пациента, записи врача на сегодня, история, статистика, экспорт.

**DoctorRepository** — поиск врачей с рейтингом (JOIN на `reviews`, AVG), поиск по имени и специализации (LIKE), возвращает типизированные объекты `DoctorProfile`.

**VisitRepository** — протоколы приёмов и назначения. Загрузка полной истории без N+1: сначала все визиты одним запросом, затем все назначения для них одним запросом (IN-список), группировка в PHP.

**ReviewRepository** — проверка уникальности отзыва, завершённые приёмы без отзыва (LEFT JOIN + IS NULL), одобренные отзывы с ответами администратора.

**AdminRepository** — агрегированные запросы для административной панели: статистика (5 COUNT-запросов), динамика за 14 дней (GROUP BY DATE), топ-5 врачей, CRUD врачей, UPSERT расписания, модерация отзывов.

### 6.5 Модели и перечисления

**DTO-модели** — иммутабельные объекты с readonly-свойствами:

```php
// User — иммутабельный объект пользователя
class User {
    public function __construct(
        public readonly int    $id,
        public readonly string $email,
        public readonly string $passwordHash,
        public readonly string $role,
        public readonly string $createdAt,
    ) {}
}
```

`DoctorProfile` — богатый DTO врача с вычисляемым `fullName`, `avgRating`, `reviewCount`, `isActive`. Гидратируется через статический фабричный метод `fromRow($row)`.

**PHP 8.1 Backed Enums** — типобезопасные перечисления для доменных понятий:

```php
enum AppointmentStatus: string {
    case Pending    = 'pending';
    case Confirmed  = 'confirmed';
    case InProgress = 'in_progress';
    case Completed  = 'completed';
    case Cancelled  = 'cancelled';
}

enum Role: string {
    case Patient = 'patient';
    case Doctor  = 'doctor';
    case Admin   = 'admin';
}

enum PrescriptionType: string {
    case Drug      = 'drug';
    case Procedure = 'procedure';
    case Referral  = 'referral';
}
```

## 7. База данных

### 7.1 Схема данных

База данных содержит 13 таблиц:

| Таблица | Назначение | Ключевые поля |
|---------|-----------|--------------|
| `users` | Единая таблица авторизации | id, email, password_hash, role ENUM('patient','doctor','admin') |
| `patients` | Профили пациентов | user_id, ФИО, birth_date, phone, gender, address, chronic_diseases |
| `doctors` | Профили врачей | user_id, ФИО, specialization_id, bio, photo_url, is_active |
| `specializations` | Справочник специализаций | id, name, description, image_url |
| `schedules` | Рабочее расписание врачей | doctor_id, day_of_week (1–7), start_time, end_time, slot_duration_min |
| `schedule_exceptions` | Отпуска и исключения | doctor_id, exception_date, is_day_off, note |
| `appointments` | Записи пациентов | patient_id, doctor_id, appointment_type, scheduled_at, status, active_slot |
| `visits` | Протоколы приёмов (1:1 к appointments) | appointment_id, started_at, ended_at, complaints, examination, diagnosis |
| `prescriptions` | Назначения врача | visit_id, type, name, dosage, notes |
| `reviews` | Отзывы пациентов | patient_id, doctor_id, appointment_id, rating, review_text, is_approved, admin_reply |
| `services` | Прайс-лист услуг | name, price, specialization_id, description |
| `lab_tests` | Каталог анализов | name, category, price, duration_min, preparation |
| `articles` | Статьи блога | slug, title, body, category, is_published, author_id |

### 7.2 Связи между таблицами

Ключевой паттерн схемы — **разделение авторизации и профилей**: таблица `users` хранит только учётные данные (email, пароль, роль), расширенные профили пациентов и врачей находятся в отдельных таблицах `patients` и `doctors` со связью 1:1 через `user_id`.

```
users ──────────────► patients (1:1, CASCADE)
users ──────────────► doctors  (1:1, CASCADE)

specializations ←──── doctors  (N:1, RESTRICT)
specializations ←──── services (N:1, SET NULL)

doctors ────────────► schedules           (1:N, CASCADE)
doctors ────────────► schedule_exceptions (1:N, CASCADE)

patients + doctors ─► appointments (N:M посредник, CASCADE)
lab_tests ──────────► appointments (опционально, SET NULL)

appointments ───────► visits        (1:1, CASCADE)
visits ─────────────► prescriptions (1:N, CASCADE)

appointments ───────► reviews (1:1, UNIQUE, CASCADE)
```

### 7.3 Индексы и оптимизация

Таблица `appointments` имеет наибольшее количество индексов, отражая её центральную роль:

- Составные индексы `(doctor_id, status)` и `(patient_id, status)` — покрывают частые запросы "записи врача в статусе X".
- Составной индекс `(status, scheduled_at)` — для фильтрации по статусу с сортировкой по дате.
- Уникальный индекс `(doctor_id, active_slot)` на сгенерированном столбце — защита от двойного бронирования (подробнее в разделе «Инженерные решения»).

Дополнительные индексы:
- `reviews(doctor_id, is_approved)` — для частого запроса «одобренные отзывы врача».
- `schedule_exceptions(doctor_id, exception_date)` — проверка исключений при генерации слотов.
- `doctors(specialization_id)` — фильтрация врачей по специализации.

CHECK-ограничения: `chk_reviews_rating` (rating BETWEEN 1 AND 5), `chk_schedules_times` (start_time < end_time).

### 7.4 Полиморфная таблица appointments

Таблица `appointments` хранит два типа записей через поле `appointment_type ENUM('doctor', 'lab_test')`:
- Запись к врачу: `doctor_id` заполнен, `lab_test_id` равен NULL.
- Запись на анализ: `doctor_id` равен NULL, `lab_test_id` заполнен.

Такое решение упрощает отчётность: история пациента, статистика и экспорт работают через единый запрос к одной таблице.

## 8. Безопасность

### 8.1 CSRF-защита

Реализована на двух уровнях (defence in depth):

1. **Глобальный middleware** — `CsrfMiddleware::handle()` применяется ко всем POST-запросам автоматически. Сравнивает `$_POST['csrf_token']` с токеном в сессии.
2. **Повторная проверка в контроллерах** — `BaseController::validateCsrf()` вызывается в POST-обработчиках дополнительно.

Каждая HTML-форма содержит скрытое поле:
```html
<input type="hidden" name="csrf_token"
       value="<?= View::e(Session::generateCsrfToken()) ?>">
```

CSRF-токен — 64-символьная hex-строка, полученная через `bin2hex(random_bytes(32))`. Сравнение через `hash_equals()` исключает тайминг-атаку на строчное сравнение.

### 8.2 Защита от XSS

Весь вывод пользовательских данных в шаблонах проходит через `View::e()`:
```php
public static function e(mixed $value): string {
    return htmlspecialchars((string)$value, ENT_QUOTES | ENT_SUBSTITUTE, 'UTF-8');
}
```
`ENT_QUOTES` экранирует одинарные и двойные кавычки, `ENT_SUBSTITUTE` заменяет невалидные байты. Данные для JavaScript-графиков передаются через `json_encode()` с последующим `htmlspecialchars()`.

Дополнительный уровень: HTTP-заголовок `Content-Security-Policy` ограничивает исполнение скриптов только из `'self'`.

### 8.3 Защита от SQL-инъекций

Все запросы используют PDO prepared statements с реальными параметрами:
```php
$stmt = $this->db->prepare("SELECT * FROM users WHERE email = ? LIMIT 1");
$stmt->execute([$email]);
```
Параметр `PDO::ATTR_EMULATE_PREPARES = false` гарантирует, что подстановка параметров происходит на стороне MySQL, а не PHP — это полностью исключает SQL-инъекции.

### 8.4 Аутентификация и хеширование паролей

- Пароли хранятся через `password_hash($password, PASSWORD_DEFAULT)` — bcrypt с автоматическим выбором алгоритма.
- При входе проверяется `password_needs_rehash()` — устаревший хеш обновляется прозрачно.
- Защита от тайминг-атаки на перечисление email: при несуществующем адресе выполняется фиктивный `password_verify()` для выравнивания времени ответа.
- `session_regenerate_id(true)` при успешном входе — защита от Session Fixation.

### 8.5 Безопасные куки сессии

```php
session_set_cookie_params([
    'lifetime' => 0,
    'secure'   => !empty($_SERVER['HTTPS']),
    'httponly' => true,
    'samesite' => 'Strict',
]);
```
- `httponly` — кука недоступна JavaScript (защита от XSS-кражи сессии).
- `samesite=Strict` — кука не отправляется в cross-site запросах (дополнительная CSRF-защита).
- `secure` — только по HTTPS в продакшене.

### 8.6 Rate Limiting

`RateLimitMiddleware` ограничивает количество запросов к критичным маршрутам на основе счётчиков в сессии:
- `POST /login` — 5 попыток за 15 минут.
- `POST /register` — 5 попыток за 1 час.
- `POST /contact` — 5 запросов за 1 час.

При превышении лимита выводится сообщение с оставшимся временем ожидания и происходит редирект назад.

### 8.7 HTTP Security Headers

Установлены в точке входа `public/index.php`:
- `X-Frame-Options: DENY` — запрет встраивания в iframe (clickjacking).
- `X-Content-Type-Options: nosniff` — запрет MIME-sniffing.
- `Referrer-Policy: strict-origin-when-cross-origin`.
- `Content-Security-Policy` — ограничение источников скриптов, стилей, шрифтов.

### 8.8 Проверка принадлежности ресурса

`OwnerMiddleware` проверяет, что запрошенный ресурс принадлежит текущему пользователю. Для маршрута печати визита выполняется SQL-запрос через JOIN `visits → appointments → patients → users` перед тем, как запрос достигнет контроллера. Пациент не может просмотреть визиты других пациентов.

## 9. UI/UX

### 9.1 Дизайн-система

Дизайн построен на системе токенов — CSS Custom Properties, определяющих единые значения цветов, отступов, скруглений и шрифтов. Это обеспечивает визуальную согласованность всего интерфейса.

**Цветовая схема:** монохромная основа (белый / серые тона) с акцентным бирюзовым (teal) для основных действий и янтарным (amber) для акцентов. Семантические цвета: зелёный (success), жёлтый (warning), красный (danger), синий (info).

**Типографическая иерархия:** три шрифта с чёткими ролями — Inter Tight для UI-текста, Instrument Serif для акцентных заголовков публичного сайта, JetBrains Mono для числовых данных, времени и меток.

### 9.2 Паттерн PRG (Post-Redirect-Get)

Все POST-обработчики после выполнения действия устанавливают flash-сообщение и выполняют HTTP-редирект. Пользователь видит результат на следующей GET-странице. Преимущество: обновление страницы не вызывает повторную отправку формы.

Flash-сообщения существуют ровно до первого показа — `Session::getFlash()` читает и одновременно удаляет сообщение из сессии.

### 9.3 Навигация по ролям

У каждой роли своя навигационная панель с соответствующими пунктами:
- **Пациент:** Дашборд, Запись к врачу, Мои записи, Медкарта, Отзывы, Профиль.
- **Врач:** Расписание, История приёмов, Профиль.
- **Администратор:** Статистика, Записи, Расписание, Отзывы, Врачи, Услуги, Анализы.

Неавторизованный пользователь видит публичный сайт с navbar и кнопками «Войти» / «Записаться».

### 9.4 Формы и валидация

Формы имеют встроенную обратную связь: при ошибках валидации страница перерендеривается с сообщениями об ошибках рядом с каждым полем и сохранёнными значениями полей (`old`). Пользователь не теряет введённые данные.

Пароль скрыт от повторного отображения при ошибке (только non-sensitive поля подставляются в `old`).

### 9.5 Пустые состояния и обратная связь

Для пустых списков используется партиал `empty-state.php` с иконкой и поясняющим текстом. Статусы записей визуально различаются цветными бейджами. Недоступные временные слоты отображаются зачёркнутыми, а не скрытыми.

## 10. Основные функции

### 10.1 Онлайн-запись к врачу

Многошаговый процесс:
1. Выбор специализации из списка с иконками и количеством врачей.
2. Выбор конкретного врача с рейтингом и биографией.
3. Выбор даты из ближайших доступных рабочих дней (pills-компонент).
4. Выбор временного слота из сетки (доступные и занятые отображаются по-разному).
5. Блок подтверждения с итоговой информацией и кнопкой записи.

Каждый шаг — отдельный GET-запрос с параметрами в URL, что позволяет возвращаться на любой шаг.

### 10.2 Запись на лабораторные анализы

Аналогичный поток, но с фиксированным расписанием лаборатории: Пн–Пт, 08:00–18:00, слот 15 минут. Пациент выбирает анализ из каталога, сгруппированного по категориям, затем дату и время.

### 10.3 Ведение медицинского приёма

Жизненный цикл приёма:
```
pending/confirmed
    ↓ [Начать приём]
in_progress — создаётся запись visits, started_at = NOW()
    ↓ [Сохранить протокол / добавить назначения]
in_progress — обновляются complaints, examination, diagnosis
    ↓ [Завершить]
completed — ended_at = NOW()
```

Протокол включает: жалобы, осмотр, диагноз. Назначения трёх типов: препарат (название, дозировка, примечания), процедура, направление.

### 10.4 Медицинская карта

Пациент видит хронологическую историю всех завершённых визитов с полным протоколом и назначениями. Каждый визит можно распечатать — система генерирует страницу для печати через CSS (`@media print`).

### 10.5 Система отзывов с модерацией

- Пациент может оставить отзыв только на завершённый приём.
- Один приём — один отзыв (уникальный индекс в БД).
- Отзыв состоит из оценки 1–5 звёзд и текста (не менее 10 символов).
- После отправки отзыв ожидает одобрения администратором.
- Одобренные отзывы видны на странице врача и влияют на средний рейтинг.
- Администратор может добавить ответ от клиники.

### 10.6 Административная аналитика

Дашборд администратора отображает:
- 5 ключевых метрик: всего пациентов, всего записей, записей сегодня, ожидающих подтверждения, завершённых в текущем месяце.
- Линейный график динамики записей за последние 14 дней.
- Столбчатый график топ-5 врачей по количеству завершённых приёмов.

Дополнительно: пагинированный список всех записей с фильтрацией по статусу и дате, экспорт в CSV с BOM-заголовком (корректное открытие в Excel).

### 10.7 Управление расписанием

Администратор настраивает расписание каждого врача по дням недели: время начала и окончания, длительность слота. Поддерживаются исключения (отпуска, выходные дни). Данные сохраняются через UPSERT (INSERT ... ON DUPLICATE KEY UPDATE).

## 11. Пользовательские сценарии

### 11.1 Сценарий пациента: от регистрации до получения медкарты

**Регистрация.** Пациент открывает `/register`, заполняет: email, пароль, подтверждение пароля, фамилию, имя, отчество, дату рождения, пол. При успешной валидации создаются записи в `users` и `patients` в транзакции, после чего происходит автоматический вход и редирект на дашборд.

**Запись к врачу.** В `/patient/book` пациент последовательно выбирает специализацию, врача, дату из доступных рабочих дней и временной слот. После нажатия «Подтвердить запись» система создаёт запись со статусом `confirmed`.

**Ожидание.** На дашборде `/patient/dashboard` видны предстоящие записи. До начала приёма пациент может отменить запись через `/patient/appointments`.

**Приём.** Врач начинает приём — статус меняется на `in_progress`. Заполняет протокол, добавляет назначения, завершает приём — статус `completed`.

**Медкарта.** Пациент в `/patient/medical-record` видит полный протокол завершённого визита с диагнозом и назначениями. Нажимает «Печать» — открывается версия для печати.

**Отзыв.** В `/patient/reviews` отображаются завершённые приёмы, на которые ещё нет отзыва. Пациент выставляет оценку и пишет текст. Отзыв уходит на модерацию.

### 11.2 Сценарий врача: рабочий день

**Вход.** Врач входит через `/login`. При неактивном аккаунте (`is_active = false`) вход заблокирован.

**Дашборд.** `/doctor/dashboard` показывает: список приёмов на сегодня, статистику (всего завершённых, средний рейтинг, приёмов в этом месяце).

**Ведение приёма.** Врач открывает запись, видит данные пациента (включая хронические заболевания и историю предыдущих визитов). Нажимает «Начать приём». Заполняет жалобы, данные осмотра, диагноз. Добавляет назначения. Нажимает «Завершить приём» — все данные сохраняются транзакционно.

**История.** `/doctor/history` — архив всех завершённых приёмов с диагнозами.

### 11.3 Сценарий администратора: управление клиникой

**Дашборд.** Администратор видит ключевые метрики клиники и графики на `/admin/dashboard`.

**Добавление врача.** `/admin/doctors/create` — форма с данными нового врача. После сохранения врач может войти в систему. При необходимости — деактивация (`is_active = false`) без удаления данных.

**Настройка расписания.** `/admin/schedule` — выбор врача, настройка рабочих часов по дням недели, длительность приёма.

**Управление записями.** `/admin/appointments` — таблица всех записей с пагинацией. Фильтрация по статусу и дате. Отмена или перенос записи на другое время.

**Модерация отзывов.** `/admin/reviews` — список неодобренных отзывов. Одобрение, удаление, написание ответа от клиники.

**Экспорт.** `GET /admin/appointments/export` — скачать CSV со всеми записями за указанный период.

## 13. Инженерные решения

### 13.1 Защита от race condition при бронировании

**Проблема:** два пациента одновременно открывают страницу с доступным слотом и нажимают «Записаться». Оба видят слот как свободный. Без защиты оба попадут на один и тот же приём.

**Решение** — трёхуровневая защита:

**Уровень 1 — `SELECT ... FOR UPDATE` в транзакции.** При подтверждении бронирования открывается транзакция, и строки занятых слотов блокируются для изменения. Конкурирующий запрос ждёт освобождения блокировки.

**Уровень 2 — повторная проверка доступности слота** внутри транзакции — исключает момент между первоначальной проверкой и началом транзакции.

**Уровень 3 — сгенерированный столбец с уникальным индексом.** Это решение заслуживает отдельного описания.

В таблице `appointments` создан сгенерированный столбец:
```sql
ADD COLUMN active_slot DATETIME GENERATED ALWAYS AS (
    IF(status <> 'cancelled', scheduled_at, NULL)
) STORED;

ADD UNIQUE KEY uq_appt_doctor_active_slot (doctor_id, active_slot);
```

`active_slot` равен `scheduled_at` для активных записей и NULL для отменённых. Уникальный индекс на `(doctor_id, active_slot)` игнорирует NULL (стандартное поведение SQL) — отменённые записи не блокируют слот. Для активных записей уникальность пары (врач, время) гарантируется на уровне базы данных. Даже если уровни 1 и 2 окажутся недостаточными, база данных отвергнет вторую запись с ошибкой `PDOException` (код 23000), которая перехватывается и превращается в понятное пользователю сообщение.

### 13.2 Устойчивость к тайминг-атакам при аутентификации

При несуществующем email в базе данных вычисление хеша не происходит, что делает ответ сервера значительно быстрее, чем при существующем email с неверным паролем. Это позволяет атакующему определять существование email по времени ответа.

**Решение:** при ненайденном email выполняется фиктивный `password_verify()` с захардкоженным невалидным хешем:
```php
password_verify($password, '$2y$12$invaliddummyhashXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX');
```
Это выравнивает время ответа для существующих и несуществующих email-адресов.

### 13.3 Иммутабельные DTO

Вместо передачи ассоциативных массивов между слоями система использует readonly-объекты. `DoctorProfile::fromRow($row)` принимает массив из базы данных и возвращает типизированный объект. Преимущества: IDE-автодополнение, невозможность случайной мутации данных, явный контракт между слоями.

### 13.4 Решение проблемы N+1 запросов

При загрузке полной медицинской карты пациента наивная реализация выполняла бы N+1 запросов: один за визитами, затем по одному за назначениями каждого визита.

**Решение:** сначала одним запросом загружаются все визиты, их идентификаторы собираются в массив, затем одним запросом с IN-условием загружаются все назначения. Группировка назначений по визитам выполняется в PHP. Итого: 2 запроса вместо N+1.

### 13.5 PHP 8.1 Backed Enums для доменных понятий

Строковые константы вместо перечислений (например, `'pending'`, `'confirmed'`) имеют недостатки: возможны опечатки, нет IDE-подсказок, сложно найти все места использования. Backed Enums решают эти проблемы, добавляя строгую типизацию на уровне языка при сохранении строковых значений в базе данных.

### 13.6 UPSERT для расписания

Форма расписания администратора отправляет данные по всем семи дням недели единовременно. Вместо логики «найди существующую запись, если есть — обнови, если нет — создай», используется MySQL-конструкция `INSERT ... ON DUPLICATE KEY UPDATE`. Это атомарная операция, не требующая предварительного SELECT и устойчивая к гонкам.

### 13.7 CSS Cascade Layers

Традиционный подход к организации CSS через специфичность селекторов приводит к «войнам приоритетов», когда для переопределения стиля нужно наращивать специфичность. `@layer` позволяет объявить явный порядок приоритетов: utilities всегда перекрывают components, которые всегда перекрывают reset — независимо от специфичности селекторов внутри слоя.

## 14. Заключение

Разработанная информационная система медицинской клиники представляет собой функционально полное веб-приложение, охватывающее полный операционный цикл учреждения здравоохранения: онлайн-запись пациентов, ведение медицинских приёмов и протоколов, управление назначениями, систему отзывов с модерацией и административную аналитику.

**Технические достижения проекта:**

- **Архитектурная чистота.** Строгое разделение на слои Controller → Service → Repository → Database с иммутабельными DTO между ними обеспечивает сопровождаемость и расширяемость.
- **Комплексная безопасность.** Реализован полный спектр мер защиты: CSRF, XSS, SQL-инъекции, тайминг-атаки, session fixation, rate limiting, owner checks — для проекта без фреймворка это нетипично высокий уровень.
- **Продакшен-уровень защита от race condition.** Трёхуровневый механизм (SELECT FOR UPDATE + повторная проверка + generated column с уникальным индексом) решает задачу конкурентного бронирования надёжнее, чем многие промышленные системы.
- **Современный стек PHP 8.1.** Применены: Backed Enums, readonly properties, constructor property promotion, named arguments — язык используется по-современному.
- **Минимализм зависимостей.** Два внешних пакета. Весь инфраструктурный слой написан самостоятельно, что даёт полный контроль над кодом и отсутствие избыточных абстракций.

**Направления возможного развития:** добавление Redis-based rate limiting по IP (текущая реализация на сессиях обходится очисткой cookie), email-уведомления о записях, REST API для мобильных приложений, двухфакторная аутентификация для привилегированных ролей, аудит-лог изменений данных.

Проект демонстрирует, что хорошо структурированное PHP-приложение без фреймворка способно обеспечить тот же уровень безопасности, типизации и архитектурной дисциплины, что и решения на основе тяжёлых фреймворков, — при значительно меньшем накладном расходе и большей прозрачности реализации.
